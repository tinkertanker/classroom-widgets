## medium: Root npm scripts use Unix-only shell commands

id: fnd_sig-feat-config-7528cb5b98-efa77_b84fde3e51
category: build-release
confidence: high
triage: risk
status: open
feature: Project config package.json (feat_config_7528cb5b98)

evidence:
- package.json:12 (scripts.dev)
- package.json:28 (scripts.clean)

The root project exposes npm scripts that rely on POSIX shell behavior. On a default Windows npm environment, `./dev.sh` is not directly executable from `cmd.exe`, and `rm -rf` is not available. That makes common root commands fail for Windows contributors or Windows CI even though the package is otherwise an npm workspace project and the review environment is Windows.

recommendation:
Replace these with cross-platform Node scripts or npm-native commands. For example, route `dev` through the existing workspace script (`npm run dev -w @classroom-widgets/server`) and implement `clean` with a Node cleanup script or a cross-platform package such as `rimraf`.

test analysis:
No linked tests or CI scripts exercise the root npm lifecycle commands across supported shells.

suggested regression test:
Add a lightweight script check in CI that runs the root lifecycle commands, or at minimum runs `npm run dev:server -- --help`/a non-blocking equivalent and `npm run clean` on Windows and Linux runners.

minimum fix scope:
Update the root `package.json` scripts for `dev` and `clean`, adding a small helper dependency or Node script if needed.

repro:
On Windows with npm's default script shell, run `npm run dev` or `npm run clean`; the shell will fail to execute `./dev.sh` or resolve `rm`.

## medium: Snake can falsely die when moving into the cell just vacated by its tail

id: fnd_sig-feat-library-05cbc88809-0a06_a2ac0ad302
category: bug
confidence: high
triage: confirmed-bug
status: open
feature: Node source packages/teacher/src/features/widgets/snake (feat_library_05cbc88809)

evidence:
- packages/teacher/src/features/widgets/snake/snake.tsx:74-78 (moveSnake)
- packages/teacher/src/features/widgets/snake/snake.tsx:92-94 (moveSnake)

The self-collision check runs against the full current snake before the tail is removed for a normal non-eating move. In Snake, moving into the current tail cell is legal because that tail moves away on the same tick. This implementation marks that move as game over before it reaches the non-food branch that pops the tail, causing valid maneuvers to fail once the snake is long enough to turn into its own tail position.

recommendation:
When checking self-collision, exclude the tail segment on non-food moves, or compute whether the next head eats food first and check against the body slice that will remain after the tick.

test analysis:
No linked tests were provided for the snake widget, and there is no evidence of movement/collision tests covering tail-vacating behavior.

suggested regression test:
Render the component or extract the movement reducer, set a snake body where the next head moves into the current tail cell without eating food, advance one tick, and assert the game is not over and the snake advances normally.

minimum fix scope:
Update moveSnake collision logic in packages/teacher/src/features/widgets/snake/snake.tsx.

repro:
Grow the snake to length greater than 1, then steer so the next head position is exactly the current tail cell on a tick where no food is eaten. The game sets gameOver instead of allowing the move.

## low: New food can spawn on the snake head immediately after eating

id: fnd_sig-feat-library-05cbc88809-deb0_1e1dd60cec
category: bug
confidence: high
triage: confirmed-bug
status: open
feature: Node source packages/teacher/src/features/widgets/snake (feat_library_05cbc88809)

evidence:
- packages/teacher/src/features/widgets/snake/snake.tsx:27-38 (generateRandomFood)
- packages/teacher/src/features/widgets/snake/snake.tsx:84-90 (moveSnake)

After the snake eats, newSnake includes the new head at the consumed food cell, but generateRandomFood filters against the React state variable snake captured by the callback, which is the pre-move snake. That old snake does not include the newly occupied head cell, so the random generator can choose the just-eaten cell and render food inside the snake.

recommendation:
Generate food against the next snake body after unshifting the head, for example by passing newSnake into a helper that excludes all occupied cells.

test analysis:
No linked tests were provided, and this edge case requires controlling random food placement after an eat event.

suggested regression test:
Mock Math.random to return the just-eaten head coordinates after an eat, advance the game one tick, and assert the next food position is not on any snake segment.

minimum fix scope:
Change food generation in packages/teacher/src/features/widgets/snake/snake.tsx to use the updated snake positions.

repro:
On a tick where the snake eats food, have Math.random produce coordinates equal to the consumed food/head cell. generateRandomFood accepts that coordinate because it only checks the pre-move snake array.

## low: Audio playback failures are unhandled

id: fnd_sig-feat-cli-command-77f75a2628-_c21e2a263f
category: bug
confidence: medium
triage: risk
status: open
feature: Node source packages/teacher/src/features/widgets/trafficLight (feat_cli-command_77f75a2628)

evidence:
- packages/teacher/src/features/widgets/trafficLight/trafficLight.tsx:39-45 (handleLightClick)

HTMLMediaElement.play() returns a Promise that can reject when audio playback is blocked, the asset cannot load, or the browser refuses playback. The rejection is ignored, producing an unhandled promise rejection in affected browsers while the UI continues updating. This is noisy in production and can fail tests or error monitoring if unhandled rejections are treated as failures.

recommendation:
Catch and intentionally ignore or log playback failures, for example by storing the play promise and attaching .catch(() => undefined) while keeping the state update independent.

test analysis:
No linked tests are included, and there is no coverage for rejected audio playback promises.

suggested regression test:
Add a component test that mocks Audio.prototype.play to reject, clicks a light, asserts the message still changes, and verifies no unhandled rejection escapes.

minimum fix scope:
Update handleLightClick in packages/teacher/src/features/widgets/trafficLight/trafficLight.tsx and add a focused test for rejected playback.

repro:
Mock Audio.prototype.play to return Promise.reject(new Error('blocked')), click a light, and observe an unhandled rejection.
